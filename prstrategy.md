# PR Стратегия для проекта BeanStore

---

## Что такое PR

**Pull Request (PR)** — механизм для предложения изменений в кодовой базе.  
Позволяет:  

- Просматривать изменения  
- Обсуждать код  
- Проводить код-ревью  
- Тестировать перед слиянием  

---

## Структура веток

-для проект— продакшн, стабильная версия  
- dev — интеграционная ветка для новых фич  
- feature/название — для отдельных задач  
- bugfix/название — для исправления ошибок  
- hotfix/название — для срочных правок на продакшене  

---

## Процесс работы с PR

### Этап 1. Создание задачи

- GitHub Issues / Trello / Jira  
- Пример: "Добавить кнопку сброса фильтров"  

### Этап 2. Создание ветки

```bash
git checkout dev
git pull origin dev
git checkout -b feature/reset-filters

Этап 3. Разработка
 • Пишем код
 • Следуем стилю проекта
 • Тестируем локально

Этап 4. Создание PR

git add .
git commit -m "feat: добавить кнопку сброса фильтров"
git push origin feature/reset-filters

 • Заполняем заголовок и описание
 • Назначаем ревьюера

Этап 5. Code Review
 • Ревьюер проверяет:
 • Соответствие задаче
 • Качество кода
 • Отсутствие багов
 • Влияние на существующий функционал

Этап 6. Правки
 • Исправляем замечания
 • Пушим новые коммиты
 • PR автоматически обновляется

Этап 7. Слияние
 • После одобрения — Squash and Merge в dev

Этап 8. Удаление ветки
 • Удаляем feature-ветку после мержа

⸻

Контрольный список для PR
 • Код работает локально без ошибок
 • Изменения не ломают существующий функционал
 • Название ветки соответствует задаче
 • Коммиты имеют понятные сообщения
 • Нет закомментированного кода или console.log
 • Обновлена документация (если нужно)

⸻

Шаблон описания PR

Описание
Краткое описание изменений

Связанная задача
Closes #123

Тип изменений
 • Новая функциональность
 • Исправление бага
 • Рефакторинг
 • Документация

Как тестировать?
 • Открыть страницу
 • Выполнить действия
 • Проверить результат

Проверено
 • Локально
 • На разных разрешениях экрана

⸻

Правила именования коммитов
 • feat — новая функциональность
 • feat: добавить фильтр по обжарке
 • fix — исправление бага
 • fix: исправить ползунок цены
 • refactor — рефакторинг
 • docs — документация
 • style — стили
 • test — тесты

⸻

Роли
 • Автор — пишет код, создаёт PR, исправляет замечания
 • Ревьюер — проверяет код, оставляет комментарии, одобряет
 • Тимлид — утверждает слияние в main, следит за процессом

⸻

Временные рамки
 • Создание PR — 5 минут
 • Первый ответ ревьюера — 24 часа
 • Исправление замечаний — 48 часов
 • Слияние после одобрения — 4 часа

⸻

Частые ошибки
 • Слишком большой PR — разбить на несколько маленьких
 • Нет описания — использовать шаблон
 • Коммиты без смысла — писать понятные сообщения
 • Не протестировано — всегда тестировать локально
 • Конфликты с dev — регулярно подливать dev в свою ветку

⸻

Пример для BeanStore
 • Задача: "Добавить кнопку сброса фильтров"
 • Ветка: feature/reset-filters
 • Код: script.js и styles.css
 • Создан PR с описанием
 • Назначен ревьюер
 • Ревьюер оставил 2 замечания
 • Автор исправил
 • Ревьюер одобрил
 • PR смержен в dev
 • Ветка удалена

⸻

Настройки репозитория GitHub
 • Require pull request before merging
 • Require approvals (1 ревьюер)
 • Dismiss stale pull request approvals
 • Require conversation resolution before merging
 • Automatically delete head branches after merge

⸻

Заключение

PR стратегия для BeanStore обеспечивает:
 • Контроль качества кода
 • Прозрачность изменений
 • Обучение команды через код-ревью
 • Стабильность ветки main
 • Быстрое отслеживание и исправление ошибок
